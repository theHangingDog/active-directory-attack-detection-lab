Resource Based Constrained Delegation


Detection & Telemetry (Resource-Based Constrained Delegation)

  1. 1.

    1. Overview

Attack Vector: The attacker (naruto_uzumaki07), possessing GenericAll rights over DC01$, configured Resource-Based Constrained Delegation to take over the Domain Controller. The attack involved creating rogue computer accounts (fakepc$ via Impacket and attackerpc$ via StandIn), writing their SIDs into the msDS-AllowedToActOnBehalfOfOtherIdentity attribute of DC01$, and abusing the S4U Kerberos extensions to forge Domain Admin tickets.

Key Log Sources:

  • Sysmon Event ID 1: Process Creation (Captures local tool execution like StandIn and Rubeus).

  • Event ID 4741: A computer account was created (Captures the rogue machine account creation).

  • Event ID 5136: A directory service object was modified (The "Smoking Gun" for the RBCD attribute write).

  • Event ID 4769: A Kerberos service ticket was requested (Captures the S4U2Proxy delegation abuse).

  1. 2.

    2. Scenario: Host-Based Tool Execution (Sysmon)

Objective: Detect the offensive tooling executed locally on the compromised Windows host (WRS01) during Scenario 2.

Logic: Attackers executing this locally must drop binaries to create the fake computer account and request the tickets. We monitor Sysmon Event ID 1 for known offensive tools executed by the compromised user.

KQL Query:

Sysmon_Events

| where EventID == 1

| where TimeGenerated > ago(12h)

| where Computer == "WRS01.shinobi.local"

| project TimeGenerated, CommandLine, ParentCommandLine, Image, MD5, LogonId, User

| where User has "naruto_uzumaki07"

| where ParentCommandLine has_any ("cmd.exe", "powershell.exe")

Analysis:

  • Actor: shinobi\naruto_uzumaki07

  • Execution Chain: The Sysmon logs perfectly narrate the local exploitation phase:

  1. StandIn_v13_Net45.exe --computer attackerpc --make (Creating the fake PC).

  1. Rubeus.exe hash /domain:shinobi.local /user:attackerpc$... (Extracting the RC4 hash).

  1. Rubeus.exe s4u /user:attackerpc$... (Executing the S4U delegation attack).

  1. Conclusion: This provides high-fidelity endpoint telemetry proving deliberate intent to execute RBCD natively from the workstation.

  1. 3.

    3. Scenario: Detecting Rogue Computer Account Creation

Objective: Detect the attacker abusing the MachineAccountQuota to create a fake computer account from a standard user context. Logic: We monitor Event ID 4741 (Computer Account Created). Standard users like naruto_uzumaki07 should not typically provision computer accounts.

KQL Query:

WindowsSecurityEvents

| where TimeGenerated > ago(12h)

| where EventID == 4741

| where SubjectUserName != "DC01$" and SubjectUserName != "Administrator"

| project TimeGenerated, EventID, Activity="Suspicious Computer Account Creation", Creator=SubjectUserName, NewComputer=TargetUserName

Analysis:

  • Creator: naruto_uzumaki07

  • NewComputer: The query successfully caught both attackerpc$ (from StandIn) and fakepc$ (from Impacket).

  • Conclusion: This event flags the vital preparation phase. Creating a computer object is necessary for an attacker who doesn't already control an account with a registered SPN.

  1. 4.

    4. Scenario: The "Smoking Gun" (RBCD Attribute Modification)

Objective: Detect the exact moment the attacker altered the Domain Controller's delegation rights. Logic: When impacket-rbcd or PowerView writes the new Security Descriptor to the target object, Active Directory generates Event ID 5136. We filter explicitly for the target attribute.

KQL Query:

WindowsSecurityEvents

| where TimeGenerated > ago(12h)

| where EventID == 5136

| where RenderedDescription has "msDS-AllowedToActOnBehalfOfOtherIdentity"

| project TimeGenerated, EventID, Activity="RBCD Attribute Modified", Actor=SubjectUserName, TargetObject=RenderedDescription

Analysis:

  • Actor: naruto_uzumaki07

  • Target Attribute: msDS-AllowedToActOnBehalfOfOtherIdentity

  • Conclusion: This is a definitive Indicator of Compromise (IoC). A standard user successfully overwriting the delegation permissions of a Domain Controller represents an active privilege escalation. The logs capture multiple instances, corresponding to both the Kali and local Windows executions.

  1. 5.

    5. Scenario: Detecting the S4U Ticket Forgery (Impersonation)

Objective: Detect the attacker utilizing the rogue computer account to forge a ticket for a Domain Admin (itachi_anbu01 or Administrator). Logic: RBCD relies on the S4U2Proxy extension to access the target. We monitor Event ID 4769 for requests targeting the Domain Controller (DC01$), specifically analyzing the Transited Services field to confirm the request routed through our known rogue machine accounts.

KQL Query:

WindowsSecurityEvents

| where EventID == 4769

| where ServiceName == "DC01$" or ServiceName has "cifs/DC01"

| where RenderedDescription has "Transited Services"

| where RenderedDescription has "fakepc" or RenderedDescription has "attackerpc"

| project TimeGenerated, EventID, Activity="RBCD S4U2Proxy Abuse", ImpersonatedUser=TargetUserName, TargetService=ServiceName, TransitedService=RenderedDescription, IpAddress

Analysis:

  • ImpersonatedUser: attackerpc$@SHINOBI.LOCAL and fakepc$@SHINOBI.LOCAL (This field can sometimes reflect the service account depending on the OS build and logging verbosity).

  • TargetService: DC01$

  • Transited Service: Explicitly lists the rogue machine accounts.

  • IpAddress: ::ffff:192.168.56.30 (Kali IP) and ::ffff:192.168.56.50 (Windows IP).

  • Conclusion: This log confirms the exploit succeeded. The attacker used the fake computer accounts to request access to the Domain Controller on behalf of another identity.

  1. 6.

    6. Defensive Recommendations (Mitigation)

To secure shinobi.local against RBCD attacks:

  1. Set MachineAccountQuota to 0: Prevent standard users from creating new computer accounts in the domain. If an attacker cannot create a fake computer, they cannot execute this attack (unless they already compromised an existing account with an SPN).

  • Fix: Edit the Default Domain Policy -> ms-DS-MachineAccountQuota -> 0.

  • Audit AD Object Permissions: RBCD requires write access (GenericAll, GenericWrite, WriteDACL, WriteOwner) over the target object. Regularly audit AD permissions using tools like BloodHound to ensure standard users like naruto_uzumaki07 do not possess excessive rights over Tier 0 assets.

  • Protected Users Group: Ensure all Domain Admins are in the Protected Users group, which prevents their credentials from being delegated via S4U.
