Setup Phase


ReadGMSAPassword Attack (Setup Phase)

  1. 1.

    1. Concept & Objective

Objective: Create and configure a Group Managed Service Account (gMSA) in the shinobi.local domain to demonstrate how misconfigured permissions can allow an attacker to retrieve the managed password.

Concept: gMSAs are a security feature meant to eliminate the need for administrators to manually manage service account passwords. The password is complex, rotates automatically, and is stored in the msDS-ManagedPassword attribute. By default, only authorized computer accounts can retrieve this password, but if a user account is granted these rights, the password can be exposed.

  1. 2.

    2. Step 1: Provisioning the Authorization Group

Objective: Create a dedicated security group that will be authorized to retrieve the gMSA password.

Tool: dsadd & net group

Commands:

dsadd group "CN=gmsa_group,DC=shinobi,DC=local" -samid gmsa_group -secgrp yes -scope g -desc "Group for gMSA access"
net group "gmsa_group" "gojo_satoru69" /add /domain
net group "gmsa_group" "WRS01$" /add /domain

Findings: We successfully created the gmsa_group. We then populated it with the user gojo_satoru69 and the workstation machine account WRS01$.

  1. 3.

    3. Step 2: Key Distribution Service (KDS) Root Key

Objective: Initialize the KDS Root Key on the Domain Controller, which is required for the DC to generate gMSA passwords.

Tool: ActiveDirectory PowerShell Module

Command:

Add-KdsRootKey -EffectiveTime ((Get-Date).AddHours(-10))

Get-KdsRootKey

Findings: The KdsRootKey was generated with Guid f9c35b68-.... Setting the effective time to 10 hours in the past bypasses the default 10-hour propagation delay, allowing immediate use of the gMSA.

  1. 4.

    4. Step 3: Creating the gMSA Account

Objective: Create the actual gMSA object and define who is allowed to retrieve its password.

Tool: New-ADServiceAccount

Command:

New-ADServiceAccount -Name "Ryuk_gMSA" -DNSHostName "dc01.shinobi.local" -PrincipalsAllowedToRetrieveManagedPassword "gmsa_group"

Get-ADServiceAccount "Ryuk_gMSA" -Properties PrincipalsAllowedToRetrieveManagedPassword

Findings: The account Ryuk_gMSA$ was successfully created. The PrincipalsAllowedToRetrieveManagedPassword attribute is correctly set to our gmsa_group, which contains our target user.

  1. 5.

    5. Step 4: Configuring Permissions & SPN

Objective: Assign necessary management rights to the gMSA and register a Service Principal Name (SPN).

Tool: Add-ADGroupMember & setspn

Commands:


Add-ADGroupMember -Identity "Remote Management Users" -Members "CN=Ryuk_gMSA,CN=Managed Service Accounts,DC=shinobi,DC=local"
setspn -a myadlab/wrs01.shinobi.local shinobi.local\Ryuk_gMSA

Findings: 1. Ryuk_gMSA was added to the "Remote Management Users" group, granting it lateral movement capabilities. 2. A custom SPN (myadlab/wrs01.shinobi.local) was registered to the account, making it a target for Kerberoasting in later modules.

  1. 6.

    6. Step 5: Endpoint Installation (Verification)

Objective: Verify that the authorized host (WRS01) can successfully install and communicate with the gMSA.

Tool: Install-ADServiceAccount

Command:

Install-ADServiceAccount -Identity "Ryuk_gMSA"
Test-ADServiceAccount -Identity "Ryuk_gMSA"

Findings: The test returned True, confirming that the host WRS01 has successfully retrieved the password from the DC and the gMSA is ready for use.
