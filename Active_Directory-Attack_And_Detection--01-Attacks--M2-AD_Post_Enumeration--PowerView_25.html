  1. 1.

    1. Hypothesis & Access Verification

Scenario: Following the successful compromise of the user kaneki_ghoul25 (Credentials found in AD Description: TokyoGhoul#1), we initiated post-exploitation reconnaissance to verify our access level and identifying the user's group memberships.

  • Command: Get-DomainUser -Identity kaneki_ghoul25

  • Findings: Confirmed access as kaneki_ghoul25. The user is a member of the "GENIN-USERS" group and has a valid SID. This access allows us to query the Domain Controller for information about other objects.

  1. 2.

    2. Target Acquisition: Domain Admins

Objective: Identify the "Keys to the Kingdom"—the specific accounts that hold Domain Admin privileges. These are our primary targets for Credential Dumping.

  • Command: Get-DomainGroupMember -Identity "Domain Admins"

  • Findings: Successfully enumerated 4 Domain Admins:
  • Administrator (Built-in)

  • madara_uchiha91

  • gojo_satoru69

  • itachi_anbu01

  1. 3.

    3. Attack Surface Mapping: Kerberoasting Candidates

Objective: Identify Service Accounts with registered Service Principal Names (SPNs). These accounts are vulnerable to Kerberoasting, allowing us to request their TGS tickets and crack them offline.

  • Command: Get-DomainUser -SPN | Select-Object samaccountname,serviceprincipalname

  • Findings: Discovered 3 potential targets:
  • svc_saiyan

  • svc_sharingan (Running MSSQL)

  • svc_deathnote

  1. 4.

    4. Infrastructure Analysis: Unconstrained Delegation

Objective: Find computers configured with Unconstrained Delegation. If a Domain Admin connects to such a machine, their TGT is stored in memory, allowing us to steal it and impersonate them (Unconstrained Delegation Attack).

  • Command: Get-DomainComputer -Unconstrained

  • Findings: The Domain Controller DC01 is configured with Unconstrained Delegation. While standard for DCs, this confirms that compromising DC01 yields total domain dominance.

  1. 5.

    5. Data Discovery: Network Share Hunting

Objective: Scan the network for non-standard file shares that might contain sensitive data (backups, config files) accessible to our low-level user.

  • Command: Get-NetShare -ComputerName DC01

  • Findings:
  • Standard Shares: ADMIN$, C$, SYSVOL.

  • Suspicious Share: Found a custom share named Shinobi_backup. This is a high-priority target for data exfiltration.

  1. 6.

    6. Lateral Movement Feasibility Check

Objective: Determine if kaneki_ghoul25 has Local Administrator rights on the Domain Controller to execute attacks like PsExec or WMI.

  • Command: Test-AdminAccess -ComputerName DC01

  • Findings: IsAdmin: False. The current user does not have administrative rights on the DC. We must rely on privilege escalation (like Kerberoasting) rather than direct lateral movement.

  1. 7.

    7. ACL Escalation Check (Managed Groups)

Objective: Check if kaneki_ghoul25 is the "Manager" of any security groups, which would allow us to modify group membership (Add Self) to escalate privileges.

  • Command: Get-DomainManagedSecurityGroup

  • Findings: No managed groups were found for this user. This closes off the "ACL Abuse" vector for this specific account, redirecting our focus back to the "Shinobi_backup" share and Kerberoasting.
