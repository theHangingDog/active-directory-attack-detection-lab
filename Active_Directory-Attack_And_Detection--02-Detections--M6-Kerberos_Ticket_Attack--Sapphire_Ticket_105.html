Sapphire Ticket


Detection & Telemetry (Sapphire Ticket Evasive Persistence)

  1. 1.

    1. Overview

Attack Vector: The attacker utilized a compromised account (madara_uchiha91) to perform a DCSync attack, exfiltrating the krbtgt AES keys. To bypass modern PAC signature validation, they initiated an S4U2self request to the KDC, forcing it to generate a perfectly signed PAC for the high-privileged user. This pristine PAC was extracted and embedded into a forged TGT signed with the stolen krbtgt key. The evasive ticket was then weaponized to access administrative shares on the Domain Controller.

Key Log Sources:

  • Security Event ID 4662: Directory Service Object Access (Detects the prerequisite krbtgt key theft).

  • Security Event ID 4769: Kerberos Service Ticket Request (Detects the anomalous S4U2self request used to generate the PAC).

  • Security Event ID 4624: Successful Logon (Detects the weaponization and lateral movement).

  • Security Event ID 5140: Network Share Object Access (Tracks access to Tier 0 administrative shares).

  1. 2.

    2. Scenario: Detecting the Prerequisite (DCSync for krbtgt)

Objective: Identify the extraction of the Active Directory master keys. Without the krbtgt key, the attacker cannot sign the final forged TGT's outer shell. Logic: We monitor Event ID 4662 for "Control Access" (0x100) requests against the directory service that involve replication GUIDs, filtering out legitimate Domain Controller machine accounts (S-1-5-18 and accounts ending with $).

KQL Query: 

WindowsSecurityEvents
| where EventID == 4662
| where ObjectServer == "DS"
| where RenderedDescription has "0x100"
| where RenderedDescription has_any ("1131f6ad-9c07-11d1-f79f-00c04fc2dcd2", "1131f6aa-9c07-11d1-f79f-00c04fc2dcd2")
| where SubjectUserSid != "S-1-5-18"
| where SubjectUserName !endswith "$"
| project TimeGenerated, SubjectUserName, Computer, Activity="krbtgt Credential Theft (DCSync)"

Analysis:

  • Actor Identification: The telemetry captures multiple DCSync attempts initiated by the compromised madara_uchiha91 account targeting DC01.shinobi.local.

  • Significance: Since the Sapphire ticket itself is incredibly stealthy, detecting the prerequisite DCSync is the most reliable "loud" indicator available to defenders before the attacker establishes deep persistence.

  1. 3.

    3. Scenario: The "Smoking Gun" (Detecting Anomalous S4U2self Requests)

Objective: Detect the exact moment the attacker tricks the KDC into legitimately generating the highly privileged PAC. Logic: The Sapphire Ticket creation relies on the S4U2self Kerberos extension. The KDC logs an Event 4769 when this service ticket is requested. We hunt for requests targeting high-value Domain Admin accounts originating from unauthorized or unexpected workstation IPs.

KQL Query:

WindowsSecurityEvents
| where EventID == 4769
| where IpAddress has "192.168.56.50"
| where TargetUserName == 'madara_uchiha91@SHINOBI.LOCAL'
| project TimeGenerated, TargetUserName, ServiceName, IpAddress, RenderedDescription

Analysis:

  • Behavioral Anomaly: The query identifies a massive spike (17 items within minutes) of service ticket requests for the high-privileged madara_uchiha91@SHINOBI.LOCAL identity.

  • Contextual Mismatch: Crucially, these requests are originating from the Kali Linux attacker machine IP (::ffff:192.168.56.50). Legitimate S4U2self delegation requests are typically performed by configured backend service accounts (like IIS web servers), not standard client subnets.

  1. 4.

    4. Scenario: Detecting Weaponization (Anomalous Network Logons)

Objective: Detect the usage of the highly evasive ticket as the attacker authenticates to Tier 0 assets.

Logic: Even though the Kerberos authentication mathematically passes all signature checks, the Domain Controller will still log the network connection (Event 4624). We monitor for administrative network logons originating from unusual source IPs.

KQL Query: 

WindowsSecurityEvents
| where EventID == 4624
| where LogonType == 3
| where TargetUserName == "madara_uchiha91"
| where IpAddress == "192.168.56.50"

Analysis:

  • Detection: The logs show 26 successful network logons (LogonType 3) for the Domain Admin account madara_uchiha91.

  • Indicator: The connections stem directly from the attacker IP 192.168.56.50 rather than a legitimate Tier 0 management workstation, exposing the forged session's origin.

  1. 5.

    5. Scenario: Tracing Lateral Movement (Administrative Share Access)

Objective: Track exactly what restricted resources the attacker accessed using the Sapphire Ticket. Logic: We monitor Event ID 5140 for access to critical administrative shares (ADMIN$, C$) by the suspect account from the identified anomalous IP.

KQL Query: 

WindowsSecurityEvents
| where EventID == 5140
| where ShareName == "\\\\*\\ADMIN$"
| where SubjectUserName == 'madara_uchiha91'
| project TimeGenerated, SubjectUserName, Computer, ShareName, IpAddress

Analysis:

  • Impact Verification: The correlation confirms that the forged identity was successfully weaponized to access the \\*\ADMIN$ share on DC01.shinobi.local.

  • Execution Link: This directly correlates to the attacker's execution of impacket-psexec, which utilizes the ADMIN$ share to upload its service executable and spawn a SYSTEM shell.

  1. 6.

    6. Defensive Recommendations (Mitigation)

  1. Protect High-Value Accounts from Delegation: Place highly sensitive accounts (like Domain Admins) into the Protected Users Security Group. Furthermore, set the Active Directory account attribute "Account is sensitive and cannot be delegated". This explicitly blocks the KDC from issuing the S4U2self tickets required to generate the PAC for the Sapphire attack.

  1. Double Reset KRBTGT: Because the Sapphire Ticket relies on the krbtgt AES key to sign the outer TGT, rotating the krbtgt password twice will instantly invalidate all existing Sapphire, Diamond, and Golden tickets in the environment.

  1. Implement Tiered Administration: Ensure that Domain Admin accounts are only allowed to log into Tier 0 assets (Domain Controllers). Set up conditional access or authentication policies that hard-block DA authentications originating from standard workstation subnets.
