Diamond Ticket


Detection & Telemetry (Diamond Ticket Attack)

  1. 1.

    1. Overview

Attack Vector: The attacker stole the krbtgt AES keys via DCSync. They then requested a legitimate TGT for a standard user (naruto_uzumaki07), decrypted the ticket offline, injected high-privileged SIDs into the Privilege Attribute Certificate (PAC), and re-encrypted it. This forged payload was injected into memory to gain undetected administrative access to network shares.

Key Log Sources:

  • Security Event ID 4662: Directory Service Object Access (Identifies the initial extraction of KDC keys).

  • Sysmon Event ID 1: Process Creation (Captures the local execution of forging tools).

  • Security Event ID 4672 / 4678: Special Privileges Assigned (Identifies standard accounts authenticating with injected admin rights).

  • Security Event ID 5140 / 5145: Network Share Access (Tracks the lateral movement of the anomalous session).

  1. 2.

    2. Scenario: Detecting the Prerequisite (DCSync for krbtgt)

Objective: Identify the unauthorized directory replication request used to steal the Active Directory keys necessary for offline ticket modification.

Logic: We monitor Event ID 4662 for Directory Service access containing the "Replicating Directory Changes" GUIDs, initiated by non-Domain Controller machine accounts.

KQL Query:

WindowsSecurityEvents
| where TimeGenerated > ago(5h)
| where EventID == 4662
| where ObjectServer == "DS"
| where RenderedDescription has "0x100"
| where RenderedDescription has_any ("1131f6ad-9c07-11d1-f79f-00c04fc2dcd2", "1131f6aa-9c07-11d1-f79f-00c04fc2dcd2")
| where SubjectUserSid != "S-1-5-18"
| where SubjectUserName !endswith "$"
| summarize Count = count() by SubjectUserName, Computer, Activity="DCSync Attempt"
| order by Count desc

Analysis:

  • Actor Identification: The telemetry successfully captured DCSync attempts originating from two distinct compromised users: naruto_uzumaki07 and madara_uchiha91 targeting DC01.shinobi.local.

  • Significance: Stealing the krbtgt encryption keys is the unavoidable "loud" step of a Diamond Ticket attack. Detecting this action allows defenders to immediately force a double password reset on the krbtgt account, neutralizing the attack path.

  1. 3.

    3. Scenario: Detecting Host-Level Forgery (Endpoint Forensics)

Objective: Detect the execution of offensive binaries utilized to automate the legitimate TGT request, perform the PAC manipulation, and inject the final payload.

Logic: We monitor Sysmon Process Creation (Event ID 1) for command-line arguments specific to the Diamond Ticket workflow.

KQL Query: 


Sysmon_Events
| where TimeGenerated > ago(5h)
| where EventID == 1
| where CommandLine has_any ("diamond", "/krbkey:", "/ticketuser:", "Rubeus")
| project TimeGenerated, Computer, User, Image, CommandLine

Analysis:

  • Execution Evidence: The logs capture the exact sequence of events executed by the attacker on WRS01.shinobi.local.

  • Timeline: 1. At 9:01:43, Rubeus.exe is executed with the diamond command, passing the stolen aes256 key to forge the payload. 2. At 9:03:30, Rubeus is executed again with the asktgs command, utilizing the forged TGT (ticket.kirbi) to request access to the Domain Controller.

  1. 4.

    4. Scenario: Behavioral Detection via Privilege Anomalies

Objective: Spot the usage of the Diamond Ticket by detecting when a standard user account suddenly authenticates with highly elevated, misaligned privileges. Logic: We hunt for Event ID 4672 / 4678 to catch standard users logging in with special administrative rights (like SeDebugPrivilege or SeTakeOwnershipPrivilege) that contradict their actual Active Directory group memberships.

KQL Query: 

WindowsSecurityEvents
| where EventID == "4672" or EventID == "4678"
| where SubjectUserName == "naruto_uzumaki07"
| where PrivilegeList has_any ("SeDebugPrivilege", "SeTakeOwnershipPrivilege")

Analysis:

  • Anomaly Verification: The query returned 4 distinct logon events for naruto_uzumaki07 where the account was granted special privileges, including SeSecurityPrivilege, on both the workstation (WRS01) and the Domain Controller (DC01).

  • Detection Value: Because naruto_uzumaki07 is a standard domain user, seeing them granted privileges reserved for local Administrators or SYSTEM immediately flags the session as compromised via a forged PAC.

  1. 5.

    5. Scenario: Tracing Weaponization & Lateral Movement

Objective: Track exactly what resources the attacker accessed using the forged Diamond Ticket. Logic: We extract the anomalous LogonId values from the privilege escalation events (Event 4672/4678) and correlate them with Network Share Object access events (Event ID 5140 / 5145).

KQL Query: 

let logonids = WindowsSecurityEvents
| where EventID in (4672, 4678)
| where SubjectUserName == "naruto_uzumaki07"
| where PrivilegeList has_any ("SeDebugPrivilege", "SeTakeOwnershipPrivilege")
| distinct SubjectLogonId;
WindowsSecurityEvents
| where SubjectLogonId in (logonids)
| summarize Count = count() by ShareName, EventID, IpAddress
| order by Count desc

Analysis:

  • Impact Mapping: The correlation reveals the attacker utilized the forged session to access highly restricted shares.

  • Targets: The query highlights multiple access events against \\*\C$ and \\*\IPC$.

  • Origin Points: The lateral movement originated from two separate IPs: 192.168.56.50 (Kali Attacker Machine) and 192.168.56.30, giving Incident Responders immediate IOCs to isolate.

  1. 6.

    6. Defensive Recommendations (Mitigation)

  1. Monitor Privileged Logons (Event 4672): Alert on special privileges assigned to any account that is not a member of a known administrative group. This is the most reliable way to catch a forged PAC.

  1. Double Reset krbtgt: To fully neutralize an active Diamond Ticket campaign, the krbtgt password must be reset twice to clear the old key history and invalidate the forged PAC signatures.

  1. Implement Kerberos Armoring (FAST): Enforce FAST to secure the initial pre-authentication phase, raising the barrier for intercepting and manipulating tickets.
