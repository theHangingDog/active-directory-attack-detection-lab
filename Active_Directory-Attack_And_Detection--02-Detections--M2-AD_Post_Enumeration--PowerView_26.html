PowerView


Detection & Telemetry (PowerView)

  1. 1.

    1. Overview

Traditional antivirus often fails to catch PowerView because it runs entirely in memory as a PowerShell script. To detect this, we utilized Script Block Logging (Event ID 4104) ingested into Microsoft Sentinel. This allows us to see the exact functions executed by the attacker (kaneki_ghoul25), even if they are obfuscated.

  1. 2.

    2. Scenario: Detecting General AD Reconnaissance

Attack: The attacker loaded the PowerView module and began mapping the domain structure (Users, Computers, Groups).

Logic: We query the Event table for Event ID 4104 containing known PowerView function names.

KQL Query:

Code snippet

Event

| where EventLog == "Microsoft-Windows-PowerShell/Operational"

| where EventID == 4104

| where RenderedDescription has_any ("Get-DomainUser", "Invoke-ACLScanner", "Get-NetGPO")

Analysis: The query returned numerous hits, confirming that kaneki_ghoul25 loaded the PowerView.ps1 module and executed multiple enumeration commands in rapid succession.

  1. 3.

    3. Scenario: Detecting Kerberoasting Preparation

Attack: The attacker searched for users with Service Principal Names (SPNs) to target for offline password cracking.

Logic: We look for the specific combination of Get-DomainUser AND the -SPN switch. This is a high-fidelity indicator of intent.

KQL Query:

Code snippet

Event | where EventID == 4104

| where RenderedDescription has "Get-DomainUser" and RenderedDescription has "-SPN"

Analysis: Successfully detected kaneki_ghoul25 executing the SPN query. This alert allows the SOC to intervene before the attacker can request TGS tickets.

  1. 4.

    4. Scenario: Detecting Share Hunting (Data Exfiltration)

Attack: The attacker ran Invoke-ShareFinder or Get-NetShare to identify non-standard file shares.

Logic: We use a two-layered detection approach:

  1. PowerShell (Event 4104): Detecting the Invoke-ShareFinder command string.

  1. Windows Security (Event 5140): Detecting a burst of access attempts to IPC$ or ADMIN$ shares.

KQL Query (Script Block):

Code snippet

Event | where EventID == 4104 | where RenderedDescription has_any ("Invoke-ShareFinder","Get-NetShare")

KQL Query (Share Access Frequency):

Code snippet

Event | where EventID == 5140 | summarize ShareAccessCount = count() by Computer

| where ShareAccessCount > 10

  1. 5.

    5. Scenario: Detecting Lateral Movement Recon

Attack: The attacker ran Test-AdminAccess to check if they had Local Admin rights on other machines.

Logic: This command generates a specific script block log entry. In a real environment, this would also correlate with failed logons (Event 4625) on the target machines.

KQL Query:

Code snippet

Event | where EventID == 4104 | where RenderedDescription has "Test-AdminAccess"

Analysis: Detected 41 instances of this check. The attacker was actively "knocking on doors" to find a machine they could compromise.
