ReadGMSAPassword Attack


Detection & Telemetry (ReadGMSAPassword)

  1. 1.

    1. Overview

Attack Vector: The attacker utilized a compromised user account (gojo_satoru69) to retrieve the managed password of a Group Managed Service Account (Ryuk_gMSA$). The detection identifies the unauthorized LDAP query for the sensitive msDS-ManagedPassword attribute, the execution of specialized dumping tools on the endpoint, and the final lateral movement via Pass-the-Hash (PtH).

Key Log Sources:

• Security Event ID 4662: Directory Service Object Access (Detects the read of the gMSA password GUID).

• PowerShell Event ID 4104: Script Block Logging (Captures the execution of gMSA dumping cmdlets).

• Security Event ID 4624: Successful Logon (Detects the anomalous NTLM network logon using the gMSA identity).

  1. 2.

    2. Scenario: Detecting Unauthorized gMSA Attribute Read

Objective: Detect the specific query for the msDS-ManagedPassword attribute, which contains the gMSA's credential data. Logic: Legitimate gMSA password retrieval is performed automatically by authorized computer accounts. When a user account like gojo_satoru69 performs this action, it is a high-confidence indicator of credential theft. We monitor Event ID 4662 for access involving the specific Managed Password GUID.

KQL Query: 

WindowsSecurityEvents
| where EventID == 4662
| where RenderedDescription has "e362ed86-b728-0842-b27d-2dea7a9df218" // Managed Password GUID
| where SubjectUserName !endswith "$"
| project TimeGenerated, SubjectUserName, Computer, ObjectName, Activity="Unauthorized gMSA Password Read", RenderedDescription

Analysis:

  • Actor Identified: gojo_satoru69.

  • Frequency: The logs show a burst of 9 items within minutes, indicating repeated or automated attempts to dump the account secrets.

  • Target Object: %5049f257..., which corresponds to the Ryuk_gMSA$ object GUID.

  1. 3.

    3. Scenario: Detecting PowerShell Dumping Tools

Objective: Detect the execution of the GMSAPasswordReader script via deep-visibility script block logging. Logic: Even if the attacker renames the script file, PowerShell Script Block Logging (Event 4104) captures the internal cmdlets and strings used to interact with the gMSA attributes.

KQL Query:


Powershell_logs
| where RenderedDescription has_any ("Invoke-GMSAPasswordReader", "--gmsa", "msDS-ManagedPassword")
| project TimeGenerated, Computer, UserName, EventID, ScriptPath, RenderedDescription

Analysis:

  • Evidence: The telemetry captures 12 distinct script blocks being executed by gojo_satoru69.

  • File Path: The script was executed from the Downloads directory: C:\Users\gojo_satoru69\Downloads\Invoke-GMSAPasswordReader.ps1.

  • Conclusion: This provides definitive host-based proof of the attacker utilizing offensive PowerShell scripts to automate the credential dump.

  1. 4.

    4. Scenario: Detecting Pass-the-Hash Lateral Movement

Objective: Detect the attacker utilizing the stolen gMSA NTLM hash to establish a remote session on the Domain Controller. Logic: gMSAs are designed for Kerberos authentication. An NTLM network logon (Logon Type 3) initiated by a gMSA from a workstation IP is highly anomalous.

KQL Query: 

WindowsSecurityEvents
| where EventID == 4624
| where TargetUserName == "Ryuk_gMSA$"
| where AuthenticationPackageName == "NTLM"
| project TimeGenerated, Computer, TargetUserName, IpAddress, LogonType

Analysis:

  • Identity: Ryuk_gMSA$.

  • Source: 192.168.56.50 (The attacker's machine).

  • Target: DC01.shinobi.local.

  • Conclusion: This event confirms the successful weaponization of the stolen hash. The attacker gained a shell on the DC as the service account, which has "Remote Management" rights.

  1. 5.

    5. Defensive Recommendations (Mitigation)

  • Principle of Least Privilege: Audit the PrincipalsAllowedToRetrieveManagedPassword attribute for all gMSAs. Remove all user accounts and broad security groups. Only specific Computer Objects should be allowed to retrieve passwords.

  • Monitor Group Membership: Create a Sentinel alert for any user added to the gmsa_group or any group that has "ReadGMSAPassword" rights in BloodHound.

  • Disable NTLM: Where possible, restrict NTLM usage to prevent Pass-the-Hash. Forcing Kerberos would prevent the attacker from using the stolen NTLM hash via tools like Evil-WinRM.
